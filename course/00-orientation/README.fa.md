# ۰۰ — مشاهده‌پذیری با رویکرد تحقیق

[English](README.md)

## از رخداد شروع کنید، نه از ابزار

کاربر می‌گوید: «برای رزرو تلاش کردم؛ مشکلی پیش آمد. چه شد؟» هنوز نمی‌دانیم کدام سرویس، کدام پیام یا کدام رزرو درگیر بوده است. پرسش مهم این نیست که «آیا لاگ داریم؟»؛ بلکه این است: **آیا شواهد کافی برای پاسخ قابل اتکا داریم؟**

## پایش در برابر مشاهده‌پذیری

پایش به پرسش‌های از پیش شناخته‌شده پاسخ می‌دهد: سرویس بالا است؟ نرخ خطا چقدر است؟ صف در حال رشد است؟ latency از آستانه گذشت؟

مشاهده‌پذیری به تحقیق دربارهٔ پرسش‌های ناشناخته کمک می‌کند: چرا این رزرو شکست خورد؟ کدام سرویس‌ها و پیام‌ها آن را پردازش کردند؟ تأخیر کجا ایجاد شد؟

```text
هشدار: تأخیر پرداخت بالا رفته است
  → metric: افزایش از ده دقیقه قبل شروع شده
    → trace: فراخوانی provider چهار ثانیه طول می‌کشد
      → log: timeout بعد از چهار ثانیه
        → metadata: همهٔ این رخدادها متعلق به یک workflow رزرو هستند
```

پایش معمولاً نشانه را پیدا می‌کند؛ مشاهده‌پذیری به توضیح آن نشانه می‌رسد.

## مشاهده‌پذیری یک محصول یا فرمول نیست

`ELK` یک backend است، OpenTelemetry استاندارد/ابزار تولید و ارسال telemetry است، و logs/metrics/traces انواع شواهدند. داشتن هر سه به‌تنهایی سامانه را قابل‌فهم نمی‌کند؛ کیفیت زمینه، قابلیت جست‌وجو و ارتباط شواهد اهمیت دارد.

| شواهد | پرسش مناسب | نقطه قوت | محدودیت |
| --- | --- | --- | --- |
| Metric | آیا در کل سامانه چیزی تغییر کرده؟ | روند، نرخ و هشدار | علت یک درخواست را توضیح نمی‌دهد |
| Trace | عملیات چه مسیری رفت و زمان کجا مصرف شد؟ | گراف اجرا و زمان | معمولاً نقطهٔ شروع می‌خواهد |
| Log | کد در این نقطه چه تصمیمی گرفت؟ | جزئیات و exception | فهرست تخت، ساختار اجرا نیست |
| Metadata کسب‌وکاری | کدام پیام‌ها یک workflow هستند؟ | هویت پایدار workflow | جای trace را نمی‌گیرد |

در این مخزن `TraceId` گراف فنی اجراست؛ `CorrelationId` عضویت workflow؛ `EventId` هویت پیام؛ و `CausationId` پیام علت. workflow ممکن است retry، پیام زمان‌بندی‌شده یا تأیید انسانی داشته باشد و از یک trace طولانی‌تر باشد.

## نگاه .NET و frontend

.NET ابزارهای پایه را دارد: `ILogger<T>`، `Activity` و `ActivitySource`، و `Meter`. OpenTelemetry روی همین diagnostics APIها listener، sampling و exporter اضافه می‌کند. یک browser یا Next.js هم می‌تواند context را تا API .NET و سپس RabbitMQ حفظ کند.
